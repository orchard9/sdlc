# Spec: Daily Cron Digest — Query Telegram Messages and Send Email via SMTP

## Overview

Add a scheduled cron job to the `sdlc` ecosystem that queries a Telegram channel or chat for messages from the past 24 hours and delivers a formatted digest email via SMTP. This enables project teams to receive daily summaries of their Telegram activity without having to open the app.

## Problem Statement

Teams using sdlc to govern AI services often coordinate over Telegram. There is currently no way to receive a structured daily summary of Telegram activity that integrates with the sdlc toolchain. Engineers miss important messages because they are buried in high-volume channels.

## Goals

1. Provide a CLI command (`sdlc telegram digest`) that queries the Telegram Bot API for recent messages and sends a formatted digest via SMTP.
2. Support configuration of Telegram bot token, chat IDs, SMTP credentials, and recipient list in `.sdlc/config.yaml` or environment variables.
3. Produce a well-formatted email digest showing message author, timestamp, and content grouped by chat/channel.
4. Enable scheduling via system cron (`crontab`) or any cron-compatible scheduler with a single CLI invocation.
5. Emit a structured log event to `.sdlc/.runs/` for observability — consistent with existing run record patterns.

## Non-Goals

- No Telegram client UI in the web frontend.
- No two-way interaction with Telegram from the digest email.
- No persistent message database — each run queries only recent messages (window configurable, default 24h).
- No webhook-based real-time delivery — pull-based polling only.

## User Story

As a project lead using sdlc, I want to receive a daily email summary of my Telegram project channels so that I can stay informed without manually checking Telegram.

**Acceptance criteria:**
1. Running `sdlc telegram digest --dry-run` prints the formatted digest to stdout without sending email.
2. Running `sdlc telegram digest` sends the digest to configured SMTP recipients and exits 0 on success.
3. If Telegram credentials are missing or invalid, the command exits non-zero with a clear error message.
4. If SMTP credentials are missing or fail, the command exits non-zero with a clear error message.
5. A run record is written to `.sdlc/.runs/<timestamp>-telegram-digest.json` after each execution.
6. The email subject includes the date and number of messages summarized.
7. Messages older than the configured time window (default: 24h) are excluded.

## Configuration

Configuration is read from `.sdlc/config.yaml` under the `telegram` key, with environment variable overrides:

```yaml
telegram:
  bot_token: "${TELEGRAM_BOT_TOKEN}"       # required
  chat_ids:                                 # required — list of chat/channel IDs to query
    - "-1001234567890"
  smtp:
    host: smtp.example.com                  # required
    port: 587                               # default: 587
    username: "${SMTP_USERNAME}"            # required
    password: "${SMTP_PASSWORD}"            # required
    from: digest@example.com               # required
    to:                                     # required
      - team@example.com
  digest:
    window_hours: 24                        # default: 24
    subject_prefix: "[sdlc Digest]"        # default: "[sdlc Digest]"
    max_messages_per_chat: 100             # default: 100
```

Environment variables take precedence: `TELEGRAM_BOT_TOKEN`, `SMTP_USERNAME`, `SMTP_PASSWORD`, `SMTP_HOST`, `SMTP_PORT`, `SMTP_FROM`, `SMTP_TO` (comma-separated).

## CLI Interface

```
sdlc telegram digest [OPTIONS]

Options:
  --dry-run        Print digest to stdout, do not send email
  --config <path>  Use alternate config file (default: .sdlc/config.yaml)
  --window <hours> Override time window in hours (default: from config or 24)
  --chat <id>      Override chat IDs (repeatable; overrides config list)
  --json           Emit JSON run summary to stdout instead of human-readable output
  -v, --verbose    Show per-message details during processing
  -h, --help       Print help
```

## Email Format

**Subject:** `[sdlc Digest] 2026-03-03 — 42 messages across 2 chats`

**Body (plain text + HTML multipart):**

```

Period: 2026-03-02 18:00 UTC → 2026-03-03 18:00 UTC
```

## Implementation Approach

- New CLI subcommand: `crates/sdlc-cli/src/cmd/telegram.rs`
- New core module: `crates/sdlc-core/src/telegram.rs` — Telegram Bot API client, message fetch, digest struct
- SMTP sending via the `lettre` crate (already a Rust standard for email; add to Cargo.toml)
- HTTP requests to Telegram Bot API via `reqwest` (already used in sdlc-server; add to sdlc-core or sdlc-cli)
- Run record written to `.sdlc/.runs/` using existing `RunRecord` pattern
- Config deserialization extends existing `SdlcConfig` struct in `sdlc-core`

## Dependencies to Add

| Crate | Version | Purpose |
|---|---|---|
| `lettre` | `0.11` | SMTP email sending |
| `reqwest` | `0.12` | Telegram Bot API HTTP client |
| `tokio` | `1` (already present) | Async runtime for HTTP |

## Error Handling

All errors returned as `SdlcError`. Exit code semantics:
- `0` — digest sent (or dry-run complete)
- `1` — configuration error (missing required field)
- `2` — Telegram API error (auth failure, rate limit, network)
- `3` — SMTP error (auth failure, delivery failure, network)

## Security Considerations

- Credentials must never be logged or included in run records.
- Environment variables take precedence over config file to allow secrets management via secrets managers.
- Config file values supporting `${ENV_VAR}` interpolation — substituted at load time.
- TLS required for SMTP (STARTTLS on port 587 or SMTPS on port 465); plaintext SMTP rejected.

## Out-of-Scope for This Feature

- Telegram webhook-based delivery.
- Integration with sdlc-server REST API (future: `/api/telegram/digest`).
- Support for Telegram groups requiring user auth (MTProto) — Bot API only.
- Email delivery tracking or open/click analytics.
