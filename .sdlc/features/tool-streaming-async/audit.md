# Security Audit: Streaming + Async Tool Execution

## Scope

This audit covers the security surface introduced by the streaming/async tool execution path:
- `crates/sdlc-server/src/routes/tools.rs` ‚Äî streaming branch in `run_tool` handler
- `crates/sdlc-server/src/state.rs` ‚Äî four new `SseMessage` variants
- `crates/sdlc-server/src/routes/events.rs` ‚Äî `tool` SSE event dispatch
- `crates/sdlc-core/src/tool_interaction.rs` ‚Äî `load_streaming_log`, `streaming_log_path`
- `frontend/src/pages/ToolsPage.tsx` ‚Äî SSE event handler and streaming progress display
- `frontend/src/hooks/useSSE.ts` ‚Äî `onToolEvent` addition
- `frontend/src/contexts/SseContext.tsx` ‚Äî `tool` event routing
- `frontend/src/lib/types.ts` ‚Äî `ToolSseEvent`, updated status union
- `frontend/src/api/client.ts` ‚Äî union return type for `runTool`

## Findings

### F1 ‚Äî Tool name validation guards all paths ‚Äî PASS

`validate_tool_name` is called at every handler entry point, including the streaming `run_tool` handler at line 144. The function allows only ASCII alphanumerics, hyphens, and underscores ‚Äî no path traversal characters. The tool script is resolved via `sdlc_core::paths::tool_script(&root, &name)` which constructs a deterministic path under the project `.sdlc/tools/` directory. **No path traversal risk.**

### F2 ‚Äî Subprocess environment injection ‚Äî PASS

The streaming path uses the same `resolve_secrets` + `extra_env` map as the synchronous path. `SDLC_SERVER_URL` and `SDLC_AGENT_TOKEN` are injected into every tool run (both paths). The env map is explicitly constructed ‚Äî there is no shell interpolation or command string construction that could be exploited. The subprocess is launched with `tokio::process::Command` with explicit args, not via a shell. **No injection risk.**

### F3 ‚Äî NDJSON line handling ‚Äî PASS

Stdout lines from the tool process are read via `tokio::io::BufReader::lines()`. Each line is parsed as JSON via `serde_json::from_str` ‚Äî invalid lines are dropped silently (debug-logged). There is no `eval`, `exec`, or dynamic code execution of tool output. Progress lines are passed directly to SSE subscribers as opaque JSON. The frontend renders `message`/`msg` fields as plain text strings (no `dangerouslySetInnerHTML`). **No code injection risk from tool output.**

### F4 ‚Äî SSE event fan-out ‚Äî PASS

The SSE broadcast channel sends `ToolRunProgress` events to all connected subscribers. Events contain: `tool name`, `interaction_id`, and the raw JSON line. All of these are data fields ‚Äî they do not convey authorization or bypass server-side checks. A subscriber receiving a progress event for a different tool's `interaction_id` gains no capability they don't already have (all authenticated users on the same local server share the same SSE stream). This is consistent with existing SSE event design (run events, ponder events, etc.). **Acceptable for the local-server deployment model.**

### F5 ‚Äî Interaction record file writes ‚Äî PASS

`streaming_log_path` constructs the sidecar path as `.sdlc/tool-interactions/<name>/<id>.log`. Both `name` and `id` are server-generated or validated:
- `name` passes `validate_tool_name` before reaching this code
- `id` is generated by `generate_run_id()` (timestamp + random suffix, hex chars only)
Neither component can contain `..` or path separators. The log file is opened with `create(true).append(true)` ‚Äî no overwrite of arbitrary files. **No path traversal risk.**

### F6 ‚Äî Frontend SSE event filtering ‚Äî PASS

The `handleToolEvent` callback in `ToolRunPanel` filters by both `event.name === tool.name` and `event.interaction_id === streamingJobIdRef.current`. This means a stale subscriber from a previous run cannot accidentally update the current run's state. The filtering happens client-side ‚Äî this is appropriate for the local deployment model where all authenticated users share the SSE stream.

### F7 ‚Äî Frontend rendering of streaming lines ‚Äî PASS

The streaming progress display renders `String(msg)` where `msg` is extracted from the JSON line's `message` or `msg` field. No HTML is rendered unsafely ‚Äî React's JSX escapes all string content. `JSON.stringify` is not used in a way that could embed executable content in the DOM. **No XSS risk.**

### F8 ‚Äî Token exposure in AGENT_TOKEN injection ‚Äî INFORMATIONAL

`SDLC_AGENT_TOKEN` is injected as a subprocess environment variable. This is the same pattern used in the existing synchronous tool run path and in agent runs ‚Äî no new exposure. The token is a local server credential for tool-to-server calls. It is not logged to the `.log` sidecar file or emitted via SSE. **No new exposure introduced by this feature.**

### F9 ‚Äî Log file retention ‚Äî PASS

`enforce_interaction_retention` is called after each streaming run completion and caps stored records at 200. The corresponding `.log` sidecar files, however, are not explicitly pruned when their parent `.yaml` record is deleted. This is a pre-existing pattern (same as the non-streaming path for any sidecar content). **Tracked as a minor gap; not introduced by this feature.**

## Summary

No security vulnerabilities were found. The streaming execution path maintains the same security posture as the existing synchronous path: validated tool names, no shell injection, env-based secret injection, and opaque data fan-out via SSE. The frontend renders streaming output safely via React's JSX escaping.

**Verdict: PASS ‚Äî no blocking findings.**
