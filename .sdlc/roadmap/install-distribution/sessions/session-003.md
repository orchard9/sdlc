### Lia Santangelo · Release Engineering

*I just looked at your Cargo.toml. You've already done the hard part.*

The `[workspace.metadata.dist]` block is there, `release.yml` is wired, the `build-setup.yml` handles the Node.js/frontend build step. This is real engineering, not placeholder config.

The only missing pieces are repos and secrets. Without `orchard9/sdlc-releases` existing, the workflow fails at the `gh release create` step with a 404. Without `HOMEBREW_TAP_TOKEN`, the Homebrew publish job fails.

**On Linux musl vs gnu:**

Current target: `x86_64-unknown-linux-gnu`. That is dynamically linked against glibc. If this binary is built on Ubuntu 22.04 (glibc 2.35), it will not run on Ubuntu 18.04 (glibc 2.27). Switch to `x86_64-unknown-linux-musl` — fully static binary, runs on any Linux. cargo-dist handles the cross-compilation via Docker automatically when you change the target string. One-line change.

**On install-path:**

`install-path = "CARGO_HOME"` installs to `~/.cargo/bin`. Wrong for end users — most people do not have `~/.cargo/bin` in their PATH unless they are Rust developers. Change to `install-path = ["~/.local/bin"]` — the universal non-root convention for Linux/Mac. The generated install scripts will PATH-manage accordingly.

### Ray Okafor · Enterprise IT Security

*The architecture is right. The README is already making promises the release backend cannot keep.*

The README shows `curl .../sdlc-releases/releases/latest/download/sdlc-installer.sh | sh`. `sdlc-releases` does not exist yet. New contributors find that, try it, fail. Broken promise is worse than no promise.

**On secrets hygiene:**

Use fine-grained PATs, not classic PATs. Scope `GH_RELEASES_TOKEN` to `Contents: Read and Write` on `orchard9/sdlc-releases` only. Same for `HOMEBREW_TAP_TOKEN` on `orchard9/homebrew-tap` only. Do not use classic `repo` scope — too broad. Fine-grained tokens mean the blast radius is contained if one is ever compromised.

`install-updater = false` in Cargo.toml: correct. No auto-update. Required for enterprise change management approval.

`install-path` to `~/.local/bin`: agreed with Lia. `~/.cargo/bin` is Rust-developer-specific. Universal PATH conventions matter for enterprise-wide rollout.
