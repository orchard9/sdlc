# Session 1 — Project Navigator

## Context

Jordan wants a project listing that works both locally (running multiple sdlc instances
on different ports) and in the cluster (sdlc-cluster fleet on `*.sdlc.threesix.ai`).

Requirements gathered:
- Filter (text, not categories)
- Standalone hub — separate dedicated navigator instance
- Locally: personal, single-user
- Cluster: org-level listing, personal-only auth

## Team recruited

- **Maya Krishnamurthy** · Developer Portal Architect — fleet listing UX, service catalogs
- **Daniel Park** · Multi-Project Developer — end-user, runs 6-10 local instances simultaneously
- **Priya Shah** · Systems Minimalist / Skeptic — HashiCorp background, challenges UI necessity

## Problem reframing

Real friction: multiple browser tabs all say "sdlc", you don't know which is which.
Two sub-problems:
1. **Orientation** — which tab is which project?
2. **Discovery** — starting fresh, where do I navigate?

Page title fix (`sdlc — {project-name}`) solves orientation at zero cost.
Hub listing solves discovery and gives a single bookmark as the starting point.

## Architecture exploration

Three options evaluated:

**Option A — Sidecar registry:** a separate process all instances register to.
Rejected — two-process story, gets out of sync, requires remember to start.

**Option B — Path-prefix gateway:** all projects on same port via path prefix.
Rejected — breaks relative URLs in React frontend.

**Option C — Config-driven listing page (chosen):**
Hub reads a YAML config file. No live discovery, no registry.
Local: `~/.sdlc/hub.yaml`. Cluster: config auto-generated by Woodpecker reconcile pipeline.
Works everywhere, zero new infrastructure.

⚑ Decided: Option C — config-driven listing, no discovery protocol

## Hub mode design

`sdlc serve --hub` (or running from a non-project directory) starts the hub instead
of a project workspace. Hub is a first-class server mode, not a bolt-on.

⚑ Decided: Hub mode is a first-class sdlc-server mode

⚑ Decided: Client-side async health checks per card — 2s timeout, non-blocking, cards render immediately

⚑ Decided: Auto-registration — `sdlc serve` appends the project to `~/.sdlc/hub.yaml` on startup

⚑ Decided: Cluster hub lives at root domain `sdlc.threesix.ai`, personal auth only

⚑ Decided: Page title fix included (`sdlc — {project-name}`) as zero-cost companion change

## Card content designed

Each card shows:
- Project name + URL/port (from config — instant)
- Green/grey reachability dot (async client-side ping)
- Active milestone name (fetched from project instance, optional)
- Feature count (fetched from project instance, optional)
- "agent running" badge (fetched from project instance, optional)

The milestone/feature/agent data is fetched per-card after render — degrades gracefully
if the project is unreachable.

## Product Summary

### What we explored
How to build a project navigator that works both locally (multiple port-per-project instances)
and in the fleet cluster, giving Jordan one bookmark to find and navigate to any project.

### Key shifts
Moved from "build a listing" to a clear architecture: hub mode is a first-class sdlc-server
mode, config-driven (not discovery-based), with client-side health checks that don't block
render or generate load. The standalone hub model resolves the local/cluster duality cleanly.

### Implications
This is a new server mode for sdlc — `sdlc serve --hub` — plus a `~/.sdlc/hub.yaml` config
contract and a companion page title fix. Cluster support requires the Woodpecker reconcile
pipeline to emit a `hub.yaml`-format config file on each run, which is a small addition to
the existing v18 reconcile work.

### Still open
1. What data fields should appear on each project card? (milestone name + feature count confirmed in mockup — agent running badge is unconfirmed)
2. Should `sdlc serve` auto-deregister projects on shutdown, or leave them showing as "not running"?
