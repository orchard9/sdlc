Design decisions captured here

⚑ Entry point: "Inspect" button on route rows where store_only=true. Dispatch routes don't show this button — they use the existing event log instead.
⚑ Panel vs. page: Inline panel (not a separate route) — keeps context, no navigation overhead. Panel overlays the route table or replaces section body.
⚑ Two-pane layout: Payload list left, detail right. Standard mail-client pattern — works well for sequential inspection without losing list position.
⚑ Replay mechanics: Button dispatches payload body to the tool directly via a new POST /api/webhooks/{route}/replay/{id} endpoint — not re-ingesting through the public webhook URL (would bypass secret verification and create duplicate storage).
? Replay target: Does "replay" always target the tool on the route, or should you be able to pick a different tool? Simpler = always the registered tool. Flexible = dropdown. Start simple.
? Retention / pagination: store_only payloads accumulate. Need a retention policy or the list grows unbounded. Show a warning if count > 10k?
→ Not in scope for V1: bulk replay, CSV export, diff between payloads, binary/non-JSON preview.
