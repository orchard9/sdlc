## Session 1: Initial Signal Load

Bootstrapped by /sdlc-convo-mine from Discord conversation (sdlc early-user feedback session).

### Signals extracted

- **Intent is the missing layer**: Xist knows commands exist (`/sdlc-` namespace) but doesn't know when to use each one. The description is often "what it does" not "when you need it."
- **Usage examples are absent**: Knowing a command exists is not the same as knowing when to reach for it. Examples ("use this when you're stuck on a blocked feature") are the bridge.
- **Cross-implementation comparison**: Xist wants to compare how Agy, Claude, and other tools handle the same concept. This is a power-user desire — learning by comparison.
- **Vision and Architecture example**: Commands to create these exist but Xist didn't know to look for them or what they were for.

### Why this might matter

A command library without "when/why" is a reference manual, not a tool. The commands are powerful but only if you know which one to reach for at a given moment. This is especially true for the sdlc-* commands which have non-obvious boundaries (e.g., sdlc-ponder vs. sdlc-empathy vs. sdlc-guideline — all are ideation-related, but serve different purposes).

### Open questions

- Is this primarily a docs problem (enrich command descriptions) or a product problem (in-app browse/search)?
- What format works best for "when to use": a one-liner, a scenario, a concrete trigger condition?
- Should the web UI have a /commands page that shows all available commands with context?
- Can we auto-generate this from the command template descriptions already in init.rs?
- Is cross-implementation comparison (Agy vs Claude) a reasonable V1 scope or a much later nice-to-have?

### Suggested first exploration

Pick 5 commands and write the "when/why" for each. Evaluate: is a one-liner sufficient, or do you need a scenario? That will inform the format. Then decide: docs-only, or in-app catalog?
